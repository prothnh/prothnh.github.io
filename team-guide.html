Executive Overview

‚ö†Ô∏è Key Customer Feedback (July 18, 2025)

"Are we maintaining our JSPs forever for the sake of this?"

The customer has expressed significant concerns about the current approach creating an ongoing maintenance burden on JSP pages just to support the translation layer.

This guide focuses on the core technical decision facing the NJ SPIRIT Frontend Upgrade project: choosing between continuing with our current RESTful Bridge approach versus migrating to Service Layer Extraction.

The Core Question

How should we expose Struts 1.1 backend logic and Oracle data to our modern React frontend while minimizing long-term maintenance burden and development complexity?

‚ö†Ô∏è Current: RESTful Bridge

RESTful Bridge approach: Parse HTML output from JSP pages and convert to JSON APIs

  • Maintains JSP pages indefinitely
  • Three-layer development process
  • HTML parsing maintenance overhead
üìã Alternative: Service Layer Extraction

Extract business logic from Struts Actions into reusable services

  • Eliminates JSP dependency
  • Two-layer development process
  • Direct data access

The Technical Decision

Why This Decision Matters Now

The customer's evolving position has shifted from "No changes to the backend" to openness about "entertaining alternatives that involve backend code modifications" if it improves maintainability.

Originally, we implemented the RESTful Bridge to avoid backend changes. However, customer feedback reveals this creates an unwanted development complexity: "I feel like there's one more layer of development with this approach."

Real-World Development Scenarios

These scenarios demonstrate the practical impact of each approach on daily development work:

üìù Use Case 1: Adding a Checkbox to a Form

Scenario: Developer needs to add a simple checkbox field to an existing client intake form.

RESTful Bridge Workflow
  1. Modify JSP page to add checkbox field
  2. Update Struts Bridge parser to extract new field
  3. Test HTML parsing logic
  4. Update React component to use new field
  5. Test end-to-end integration

Estimated Time: 4-6 hours

Risk Points: Parser brittleness, HTML structure changes

Service Layer Workflow
  1. Update service interface and implementation
  2. Update React component to use new field
  3. Test service and React integration

Estimated Time: 2-3 hours

Risk Points: Standard API contract changes

‚öôÔ∏è Use Case 2: Business Logic Change (No UI Impact)

Scenario: Update validation rules for case eligibility (e.g., change income threshold from $50K to $55K).

RESTful Bridge Workflow
  1. Modify business logic in Struts Action
  2. Test that JSP output format hasn't changed
  3. Verify parser still extracts data correctly
  4. Update any error message parsing if needed
  5. Test React application receives correct data

Estimated Time: 3-4 hours

Risk Points: Unexpected HTML changes, parser failures

Service Layer Workflow
  1. Modify business logic in service class
  2. Test service logic directly
  3. Verify React receives correct responses

Estimated Time: 1-2 hours

Risk Points: Standard business logic testing

‚ö†Ô∏è Developer Impact Summary

The RESTful Bridge approach consistently requires developers to think about and maintain three layers (JSP + Parser + React) even for simple changes. The Service Layer approach reduces this to two layers (Service + React).

Implementation Strategy

Phase 1: Foundation (Months 1-3)

  • Set up REST controller infrastructure within existing Struts application
  • Create service interface patterns and dependency injection
  • Extract 2-3 core services as proof of concept
  • Implement comprehensive testing framework

Phase 2: Core Migration (Months 4-9)

  • Extract business logic from Struts Actions to services systematically
  • Module-by-module migration with parallel testing
  • Update React components to use new service APIs
  • Maintain backward compatibility during transition

Phase 3: Optimization (Months 10-12)

  • Complete remaining service extractions
  • Decommission JSP pages and HTML parsing logic
  • Performance optimization and monitoring
  • Team training on new development workflow
